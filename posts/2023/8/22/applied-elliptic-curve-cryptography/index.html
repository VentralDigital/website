Applied Elliptic Curve Cryptography

August 22, 2023 by patrickd

If you've spent any time trying to learn about Elliptic Curve Cryptography, you have probably already heard a lot about the theory but not that much about how it's then used in practice. At the beginning of this article, we'll quickly review those theoretical basics before moving on to some example applications.

Review

The pretty "elliptical curve" plots, as seen in the top row, are created over an infinite field (the real numbers, R\mathbb{R}R) which is useful for getting an intuition of what's going on. But the discrete log problem, required for ECC, is only present over a finite field (integers smaller than a selected prime number, Zp\mathbb{Z}_{{p}}Zp​), where this graphical representation collapses due to the points' coordinates being determined within (mod  p){\left(\text{mod }\ {p}\right)}(mod  p). The exact formulas by which these coordinates are determined differ between curves and are mostly irrelevant unless you're attempting to roll your own crypto. The points appear to be randomly distributed but can actually become elements of a cyclical group.

  • For such a group to be complete, an operation is required under which several properties hold true. In the case of ECC, this operation is usually referred to as "point addition", but it doesn't actually have anything to do with any sort of mathematical addition, rather the operation was "made up" to fit the rules.
    • A point P1{P}_{{1}}P1​ on the curve "added" to itself (P1+P1=P2{P}_{{1}}+{P}_{{1}}={P}_{{2}}P1​+P1​=P2​) is defined as an operation that draws a tangent on the curve through P1{P}_{{1}}P1​. Thanks to the elliptic curve's nature, this tangent line will cross the curve at exactly one other point, which is mirrored over the x-axis and then declared to be the "result of the addition", point P2{P}_{{2}}P2​.
    • Adding two different points together (P1+P2=P3{P}_{{1}}+{P}_{{2}}={P}_{{3}}P1​+P2​=P3​) is defined as drawing a line that goes through both points, resulting in a third point, whose mirrored point is the result P3{P}_{{3}}P3​. As implied by this process of mirroring, there'll always be two possible Y values for each X value as these curves are horizontally symmetric.
    • Note that it's not possible to multiply or divide points unless scalar values are used to do so. For example, P1+P1=P2{P}_{{1}}+{P}_{{1}}={P}_{{2}}P1​+P1​=P2​, can also be represented as 2P1=P2{2}{P}_{{1}}={P}_{{2}}2P1​=P2​. While points are usually represented by upper-case letters, scalars are lower-case, ie.: nP=P1+P2+…+Pn{n}{P}={P}_{{1}}+{P}_{{2}}+\ldots+{P}_{{n}}nP=P1​+P2​+…+Pn​.
    • Scalars are normal integers and not part of the group, which means that between scalars any usual mathematical operation is available. But they're still part of a field (mod  n){\left(\text{mod }\ {n}\right)}(mod  n) where n{n}n is the order of the cyclic group (ie. the total number of elements in the group, not the same as the prime p{p}p defining the finite field). That means that modular arithmetic is applied where calculations between scalars happen.
    • From the "addition" operation, one can also find a reverse "subtracting" operation. Here too it's important to remember that a "negative point" −P-{P}−P is simply the x-axis-mirrored point P{P}P and does not involve any actual mathematical subtraction.
  • The properties holding true for said operation are the following:
    • Closure: The result of the operation applied upon two elements that are part of the group, always yields a result that is an element of the group as well. Meaning that the "addition" of any valid points will result in another valid point on the curve.
    • Commutativity: P1+P2=P2+P1{P}_{{1}}+{P}_{{2}}={P}_{{2}}+{P}_{{1}}P1​+P2​=P2​+P1​
    • Associativity: (P1+P2)+P3=P1+(P2+P3){\left({P}_{{1}}+{P}_{{2}}\right)}+{P}_{{3}}={P}_{{1}}+{\left({P}_{{2}}+{P}_{{3}}\right)}(P1​+P2​)+P3​=P1​+(P2​+P3​)
    • Existence of an "Identity" element O{\mathcal{\text{O}}}O within the group for which the following is true: O+P1=P1{\mathcal{\text{O}}}+{P}_{{1}}={P}_{{1}}O+P1​=P1​. No such Identity Point naturally occurs on an elliptic curve, therefore this too is "made up" as a "point at infinity".
    • Invertibility: For every element such as P1{P}_{{1}}P1​ there's an inverse −P1-{P}_{{1}}−P1​ such that P1−P1=O{P}_{{1}}-{P}_{{1}}={\mathcal{\text{O}}}P1​−P1​=O.

When the order n{n}n (ie. the number of existing points) of an elliptic curve is prime, almost any non-identity point on the curve can serve as a generator point G{G}G (also referred to as "primitive element", or "base point"). A generator, when "added" to itself over and over again, will cycle through all elements of the group (G,2G,3G,…,(n−1)G,{G},{2}{G},{3}{G},\ldots,{\left({n}-{1}\right)}{G},G,2G,3G,…,(n−1)G, O,G,2G,…{\mathcal{\text{O}}},{G},{2}{G},\ldotsO,G,2G,…). However, not all curves have a prime order. For those that have a composite order, it's common to choose a large prime-order subgroup and a generator that is part of this sub-group is then determined. These generators won't iterate through all points on the curve, only through points in their subgroup. If the subgroup's order is a large prime, it will have a number of elements comparable to the entire curve.

The fact that an exponentiation with a modulus (gs  (mod  p){g}^{{s}}\ \text{ (mod }\ {p}\text{)}gs  (mod  p)) is hard to reverse is referred to as the discrete logarithmic problem (DLP). Similarly, it's easier to determine a point P{P}P by multiplying a scalar with a generator (P=sG{P}={s}{G}P=sG) than it is determining the scalar that was used from knowing both P{P}P and G{G}G, which is referred to as the discrete logarithmic problem in ECC (ECDLP). This operation can be efficiently executed forward thanks to methods such as the Double-And-Add algorithm, while the reverse requires a much more brute force approach.

Note that most scientific paper will not make use of the ECDLP scalar-point notation. Instead they typically use the conventional discrete logarithm notation where the base is the generator and the field element (scalar) is the exponent:

C=aG+bG=(a+b)G⇔c=ga⋅gb=ga+b  (mod  n){C}={a}{G}+{b}{G}={\left({a}+{b}\right)}{G}\Leftrightarrow{c}={g}^{{a}}\cdot{g}^{{b}}={g}^{{{a}+{b}}}\ \text{ (mod }\ {n}\text{)}C=aG+bG=(a+b)G⇔c=ga⋅gb=ga+b  (mod  n)

Using ECC instead of relying on the conventional DLP or the prime factorization problem (RSA) comes with benefits: In ECC the numbers used can be much smaller while offering stronger security guarantees. It can also be argued that ECC has much fewer footguns.

ECDH

The Diffie–Hellman key exchange scheme can be implemented using ECC. It's a simple and elegant example to gain a bit of intuition in how ECDLP can be made use of in practice.

Arriving at the shared Point C

In this scheme, both Alice and Bob are able to arrive at the same shared key because scalars can be multiplied with each other:

C=aB=bA{C}={a}{B}={b}{A}C=aB=bA

C=a(bG)=b(aG){C}={a}{\left({b}{G}\right)}={b}{\left({a}{G}\right)}C=a(bG)=b(aG)

C=(a⋅b)G{C}={\left({a}\cdot{b}\right)}{G}C=(a⋅b)G

Without leaking the shared key to eavesdroppers

Eve won't be able to determine C{C}C as there's no multiplication operation defined between points:

C=A⋅B\cancel{{{C}={A}\cdot{B}}}C=A⋅B

And the point you'd arrive at using addition would be a completely different one:

D=A+B=aG+bG=(a+b)G{D}={A}+{B}={a}{G}+{b}{G}={\left({a}+{b}\right)}{G}D=A+B=aG+bG=(a+b)G

Therefore the shared key C{C}C can not be derived from observing exchanged public key points.

Without disclosing private scalars to the counter-party

It's worth noting that not even Bob will be able to determine a{a}a despite knowing all other parameters:

C=(a⋅b)G{C}={\left({a}\cdot{b}\right)}{G}C=(a⋅b)G

Solving for a{a}a is not possible as Bob won't be able to calculate CG\frac{{C}}{{G}}GC​ as no such "point-division" operation is defined.

ECIES

The "Integrated Encryption Scheme" is rather similar to what you could achieve by combining ECDH with any symmetric encryption scheme such as AES. The difference is that in a ECIES implementation you won't have to bother with combining them correctly, it's already "integrated" and can make use of your usual public key pair.

Arriving at the shared secret S

Bob arrives at the secret S{S}S by multiplying the ephemeral secret key r{r}r with Alice's public key A{A}A. Alice's public key is simply her secret scalar a{a}a multiplied with the generator point G{G}G. When Alice receives the public ephemeral key R{R}R, all she has to do is mix in her private scalar a{a}a to arrive at the same point S=arG{S}={a}{r}{G}S=arG

S=rA=aR{S}={r}{A}={a}{R}S=rA=aR

S=r(aG)=a(rG){S}={r}{\left({a}{G}\right)}={a}{\left({r}{G}\right)}S=r(aG)=a(rG)

S=arG=arG{S}={a}{r}{G}={a}{r}{G}S=arG=arG

Ephemeral Key Freshness

When combining ECDH with symmetric encryption schemes such as AES, it's essential to choose an appropriate "block cipher mode of operation". This ensures that, despite the same shared key being reused, the resulting ciphertext would always be unique preventing pattern recognition or message replay attacks via a Man-in-the-Middle.

In contrast, ECIES addresses these concerns by always generating a new ephemeral key pair for each encryption operation. Each encrypted message is accompanied by a fresh public part of the ephemeral key R{R}R, allowing the recipient to compute the decryption key uniquely for each message.

Schnorr Signatures

Various protocols employ ECC for message signing, among which Schnorr's algorithm stands out as perhaps the most elegant and simplistic, both in conceptual understanding and extensibility. Regrettably, its patent hindered a broad adoption and led many open-source projects to opt for the more intricate ECDSA protocol. However, with the patent's recent expiration, this is a great opportunity to transition to the Schnorr paradigm!

As before, we assume that protocol participants have agreed upon a curve, its parameters and the generator point. Furthermore we assume that participants have public key-pairs ready and a message m{m}m they'd like to sign and verify.

Signing

  1. Generate a reliably random, one-time-use, key-pair R=kG{R}={k}{G}R=kG. To prevent leakage of the user's private key, it's essential that k{k}k is never leaked or shared, but rather discarded immediately after the signing process.
  2. Calculate
    s=k−hash(m,R)⋅a{s}={k}-\text{hash}{\left({m},{R}\right)}\cdot{a}s=k−hash(m,R)⋅a.
    With the message m{m}m and the public one-time key R{R}R being concatenated and hashed resulting in a scalar value. Little a{a}a is the private-key part of the user's personal key-pair A=aG{A}={a}{G}A=aG. This process requires no elliptic curve operations and is very efficient thanks to that.
  3. Share s{s}s and R{R}R (both values make up a complete Schnorr Signature) with a person that wants to verify that the message was truly signed by the owner of A{A}A.

Verification

The receiver merely has to verify that, with the known values R,s,G,m,A{R},{s},{G},{m},{A}R,s,G,m,A, the following holds true:

R=sG+hash(m,R)⋅A{R}={s}{G}+\text{hash}{\left({m},{R}\right)}\cdot{A}R=sG+hash(m,R)⋅A

But why does that work? To show that, we take the formula used during the signing process and multiply both sides with G{G}G:

s=k−hash(m,R)⋅a{s}={k}-\text{hash}{\left({m},{R}\right)}\cdot{a}s=k−hash(m,R)⋅a

Gs=G⋅(k−hash(m,R)⋅a){G}{s}={G}\cdot{\left({k}-\text{hash}{\left({m},{R}\right)}\cdot{a}\right)}Gs=G⋅(k−hash(m,R)⋅a)

Gs=Gk−hash(m,R)⋅aG{G}{s}={G}{k}-\text{hash}{\left({m},{R}\right)}\cdot{a}{G}Gs=Gk−hash(m,R)⋅aG

Gs=R−hash(m,R)⋅A{G}{s}={R}-\text{hash}{\left({m},{R}\right)}\cdot{A}Gs=R−hash(m,R)⋅A

Basically, by multiplying both sides with the known parameter G{G}G we are able to substitute all unknown parameters (k{k}k and a{a}a) with known ones (R{R}R and A{A}A), while keeping the formula itself equivalent to the one used during signing.

Forging

But if the formula still works with all parameters being public, doesn't that mean that someone would be able to generate a signature using only known public parameters?

Assuming G{G}G is generally known, and we want to forge a signature for the public key A{A}A, making it look like they have signed an attacker-chosen message m{m}m. The attacker would need to determine R{R}R and s{s}s, but the problem here is that the relationship of R−hash(m,R){R}-\text{hash}{\left({m},{R}\right)}R−hash(m,R) is impossible to resolve. With the hash being a deterministically random one-way-street, how do you determine an R{R}R that when hashed with the message (h=hash(m,R){h}=\text{hash}{\left({m},{R}\right)}h=hash(m,R)) yields a scalar that works for R−h{R}-{h}R−h. This is a mathematically unresolvable circular dependency. The only feasible method is guesswork – and the number of possible guesses is astronomically high, making it impractical.

Nonce Reuse

One of the well-known vulnerabilities of ECC based signatures happens by reusing the same random factor for signing multiple messages. Meaning that, at some point, someone published two signatures (R,s1){\left({R},{s}_{{1}}\right)}(R,s1​) and (R,s2){\left({R},{s}_{{2}}\right)}(R,s2​) with both having the same R{R}R, which implies that the same random k{k}k was reused for signing two different messages m1{m}_{{1}}m1​ and m2{m}_{{2}}m2​. This issue is so detrimental because by subtracting the signatures from each other we can actually determine the private key:

(s1=k−hash(m1,R)⋅a){\left({s}_{{1}}={k}-\text{hash}{\left({m}_{{1}},{R}\right)}\cdot{a}\right)}(s1​=k−hash(m1​,R)⋅a) minus (s2=k−hash(m2,R)⋅a)→{\left({s}_{{2}}={k}-\text{hash}{\left({m}_{{2}},{R}\right)}\cdot{a}\right)}\to(s2​=k−hash(m2​,R)⋅a)→

s1−s2=(k−hash(m1,R)⋅a)−(k−hash(m2,R)⋅a){s}_{{1}}-{s}_{{2}}={\left({k}-\text{hash}{\left({m}_{{1}},{R}\right)}\cdot{a}\right)}-{\left({k}-\text{hash}{\left({m}_{{2}},{R}\right)}\cdot{a}\right)}s1​−s2​=(k−hash(m1​,R)⋅a)−(k−hash(m2​,R)⋅a)

s1−s2=k−hash(m1,R)⋅a−k+hash(m2,R)⋅a{s}_{{1}}-{s}_{{2}}={k}-\text{hash}{\left({m}_{{1}},{R}\right)}\cdot{a}-{k}+\text{hash}{\left({m}_{{2}},{R}\right)}\cdot{a}s1​−s2​=k−hash(m1​,R)⋅a−k+hash(m2​,R)⋅a

s1−s2=(k−k)+(hash(m2,R)⋅a−hash(m1,R)⋅a){s}_{{1}}-{s}_{{2}}=\cancel{{{\left({k}-{k}\right)}}}+{\left(\text{hash}{\left({m}_{{2}},{R}\right)}\cdot{a}-\text{hash}{\left({m}_{{1}},{R}\right)}\cdot{a}\right)}s1​−s2​=(k−k)​+(hash(m2​,R)⋅a−hash(m1​,R)⋅a)

s1−s2=a⋅(hash(m2,R)−hash(m1,R)){s}_{{1}}-{s}_{{2}}={a}\cdot{\left(\text{hash}{\left({m}_{{2}},{R}\right)}-\text{hash}{\left({m}_{{1}},{R}\right)}\right)}s1​−s2​=a⋅(hash(m2​,R)−hash(m1​,R))

s1−s2hash(m2,R)−hash(m1,R)=a\frac{{{s}_{{1}}-{s}_{{2}}}}{{\text{hash}{\left({m}_{{2}},{R}\right)}-\text{hash}{\left({m}_{{1}},{R}\right)}}}={a}hash(m2​,R)−hash(m1​,R)s1​−s2​​=a

As you can see, k{k}k was cancelled out, leaving only the need for publicly known parameters to determine the private key a{a}a.

This is why it's very important to ensure a reliable source of randomness is used for the secret scalar k{k}k. Another way to ensure this is by using the message itself as the source of entropy to deterministically obtain a unique random k{k}k every time (RFC 6979).

Nonce as Data proof

One of Blockchain's real-world applications is "Proof of Inclusion", meaning a proof that someone (signer) knew about something (hashed) at a certain point in time. In Bitcoin this was typically done using the OP_RETURN operation which allowed storing random data, but at the same time, wasted costly blockspace. A neat way to optimize this is by exploiting k{k}k to, additionally to being a random factor, also proof some data. This can be done without influencing the normal verification process and without increasing the signature size.

We'll still use the same formula for signing (s=k−hash(m,R)⋅a{s}={k}-\text{hash}{\left({m},{R}\right)}\cdot{a}s=k−hash(m,R)⋅a) with the difference that a new parameter j{j}j will be random within k{k}k instead of k{k}k itself:

k=j+hash(data,jG){k}={j}+\text{hash}{\left({d}{a}{t}{a},{j}{G}\right)}k=j+hash(data,jG)

The R{R}R value will be calculated as usual based on the resulting k{k}k and someone who merely wants to check the signature's validity can do so normally. But once the signer wants to proof the inclusion of the specified data{d}{a}{t}{a}data, they publish it plus J{J}J (from J=jG{J}={j}{G}J=jG). Those interested in validating the proof of inclusion can then additionally verify the following:

R=J+hash(data,J)⋅G{R}={J}+\text{hash}{\left({d}{a}{t}{a},{J}\right)}\cdot{G}R=J+hash(data,J)⋅G

This equation is the result of multiplying both sides with G{G}G: k⋅G=(j+hash(data,jG))⋅G{k}\cdot{G}={\left({j}+\text{hash}{\left({d}{a}{t}{a},{j}{G}\right)}\right)}\cdot{G}k⋅G=(j+hash(data,jG))⋅G

This technique is being made use of by various concepts in Bitcoin, such as P2CH ("Pay to Contract Hash"), "Taproot" and "Graftroot" (never question bitcoin naming, it doesn't make sense most of the time). But most interestingly: It shows how vast the possibilities of ECC still are and the potential of playfully finding new schemes.

EdDSA

This is a "Digital Signature Algorithm" typically used with the Edwards-curve, which has some efficiency and security benefits. You may notice that it's very similar to Schnorr, with the biggest difference being how the random nonce k{k}k has been replaced with a deterministic secret factor r{r}r derived from the private key a{a}a and the message m{m}m being signed.

Signing

A EdDSA signature too consists of two components (R,s){\left({R},{s}\right)}(R,s):

R=rG⇐r=hash(hash(a),m){R}={r}{G}\Leftarrow{r}=\text{hash}{\left(\text{hash}{\left({a}\right)},{m}\right)}R=rG⇐r=hash(hash(a),m)

s=r+hash(R,A,m)⋅a{s}={r}+\text{hash}{\left({R},{A},{m}\right)}\cdot{a}s=r+hash(R,A,m)⋅a

As before, all inputs of hash()\text{hash}{()}hash() functions are concatenated with each other and return a scalar value. Above, the message m{m}m is signed by the owner of the key-pair A=aG{A}={a}{G}A=aG.

Verification

Using the publicly known variables G,R,A,m{G},{R},{A},{m}G,R,A,m it's now possible to validate the signature by checking if the following equation holds true:

sG=R+hash(R,A,m)⋅A{s}{G}={R}+\text{hash}{\left({R},{A},{m}\right)}\cdot{A}sG=R+hash(R,A,m)⋅A

That this indeed works, can be proven by substituting s{s}s and showing that both sides of the equation should indeed be equal:

R+hash(R,A,m)⋅A=R+hash(R,A,m)⋅A{R}+\text{hash}{\left({R},{A},{m}\right)}\cdot{A}={R}+\text{hash}{\left({R},{A},{m}\right)}\cdot{A}R+hash(R,A,m)⋅A=R+hash(R,A,m)⋅A

Forging

As we did in Schnorr's signature scheme, here too we have a paradoxical relationship that cannot be resolved: R+hash(R){R}+\text{hash}{\left({R}\right)}R+hash(R). In order to forge a signature one would have to pick an R{R}R, that when hashed, returns a result that satisfies the outside R{R}R value. This cannot be resolved neither in a mathematical or logical manner nor using brute force - at least not in a reasonable manner.

ECDSA

You've probably heard about the Elliptic Curve Digital Signature Algorithm before, or at least its abbreviation. It's widely adopted, although arguably much less elegant than Schnorr and also just a bit intimidating due to its use of modular arithmetic inverses.

Signing

Like Schnorr, ECDSA signatures require a reliably random one-use key-pair (R=kG{R}={k}{G}R=kG). The signature is made up of two components (r,s){\left({r},{s}\right)}(r,s) that proof that a message m{m}m was signed by a specific private key a{a}a.

The r{r}r value is simply the x-coordinate of the random point R{R}R:

r=Rx{r}={R}_{{x}}r=Rx​

The s{s}s value is the scalar returned by hashing the message m{m}m, which is added to the product of R{R}R's x-coordinate and the private key a{a}a. This is then multiplied with the inverse of k{k}k, which is a reminder that these equations are using modular arithmetic over (mod  n){\left(\text{mod }\ {n}\right)}(mod  n), with n{n}n being the order of the curve (number of elements in the group).

R=kG{R}={k}{G}R=kG

You may be wondering how a verifier can know R{R}R's y-coordinate, as it was never communicated. The fact that only Rx{R}_{{x}}Rx​ was communicated through r{r}r means there are two possible Ry{R}_{{y}}Ry​ to choose from due to the curve's symmetric nature. Protocols determine the correct one in different manners, some simply declare some y-values as "canonical" based on an agreed upon rule. Others, like Ethereum (v{v}v), use an extra byte as "recovery-id" to communicate which one it is.

Malleability

The use of such a "recovery-id" introduces malleability: This id can be flipped, making the verifier expect to find a signature for the mirror point −R-{R}−R and as it turns out, it's rather simple to determine the (r,s){\left({r},{s}\right)}(r,s) parameters for this point:

# Generate one-time-use key-pair.
k = secrets.randbelow(n) + 1
R = G * k
print(f"One-time-use private key ({k}) and public key ({R.x()}, {R.y()})")
 
# Signing: s = k^(-1) * (hash(m) + r*a)\
m = b"I signed this message, cheers Alice"
e = int.from_bytes(hashlib.sha256(m).digest(), 'big') % n

sm = (-s) % n

print(f"Tampered signature (additive inverse of s) ({R2i.x()}, {sm}) for message ({m})")
